## 1. 設計書を素直に読むと、4つの目的が混在している

| 設計書のシグナル | 読み取れる目的 |
|---|---|
| 「日本の石油備蓄、いまこれだけ。」をリアルタイム可視化 | **A. 公的データのダッシュボード**（情報提供） |
| Minimalデザインで「事実を静かに伝える」 | **B. 静謐な情報サイト**（読み物・ブランディング） |
| OGP/シェアボタン/タンカーセクションで注目度を取る | **C. バイラル・SEOで流入を取るメディア** |
| Built-by-AI + 横幅フルCTA → nextlab.jp/contact | **D. ネクストラボのリード獲得用ショーケース** |

設計書はこの A〜D の優先順位を示していない。**つまり何を最適化するのかが定まっていない**。

A〜D は両立しうるが、KPI と意思決定が衝突する場面が必ず来る:
- 「Minimalで静かに」(B) と 「シェアでバズらせる」(C) は文体・密度が逆
- 「公的データの厳密性」(A) と 「AIで素早く作る」(D) は精度コストが逆
- 「リアルタイム感」(C) と 「データ実体は月次」(A) は構造的に矛盾

---

## 2. 真の目的はおそらく D（リード獲得ショーケース）

行間を読むと、本サイトの**真の目的は「ネクストラボがAIプロダクトを自社で作れる証拠物件」**として機能することだろう。

根拠:
- 設計書末尾の最重要ファネル: Built-by-AI → CTAボタン → nextlab.jp/contact
- v1.0→v2.0の最大変更が「CTAをテキストリンク→横幅フル黒ボタン」に格上げ
- そもそも「石油備蓄」という具体テーマでなくてもよい（時事ネタ何でも可）
- 開発引き継ぎ資料の発行者が「株式会社ネクストラボ」

**であれば、A/B/C は D に従属する手段**として位置づけ直すべき。

> **真の問い: このサイトは、ネクストラボへの「お問い合わせ件数」と「契約獲得」をどれだけ生むか？**

この問いがフレームになって初めて、設計の優先度が決まる。

## 3. 何ができるか / できないか（一次情報検証ベース）

### ✅ できる
- 経産省「石油備蓄の現況」を**月次**で取得・表示（3区分内訳含む）
- 過去12ヶ月の備蓄日数推移グラフ
- 月次値からの**疑似リアルタイムアニメーション**（線形補間）
- OGP動的生成（@vercel/og、月次UUIDキャッシュ無効化）
- シェアボタン・X連携
- 単位換算・スケール感・豆知識の静的コンテンツ
- aboutタブ + CTAボタン
- aisstream.io（**無料**）でのタンカー周辺AIS購読（日本接近〜入港の検知）
- 政府データの**商用利用**（CC BY 4.0互換、出典明記必須）

### ❌ できない / 大幅修正が必要
- **「時分秒」の真のリアルタイム表示** — 一次データは月次・前々月末ベース、最大2ヶ月遅延
- **「300日」を基準にしたタンクゲージ** — 法定基準にも実績にもない数値
- **設計書5港でのタンカー捕捉** — 国内最大中継基地の喜入が漏れている
- **Datalastic AIS の「laden判定」「destination=日本港フィルタ」をサーバ側で** — 自前推計・正規化が必要
- **Datalastic生データのas-is表示** — 利用規約違反リスク
- **「世界第5位」「314万bbl/日」の単純表示** — 値が古い、または二重基準で整合しない

### ⚠ 不明（実機検証が必要）
- e-Stat API `getStatsData` で `00551140` がJSON取得できるか
- 競合「gontarobee」が実在するのか（検索ヒットなし）
- Datalastic規約「raw/as-is」の運用許容範囲（要サポート確認）

## 4. 筋が良い点（残す/活かすべき判断）

### 4.1 トピック選定: 石油備蓄
- 時事性が高く、**地政学イベント（中東情勢、IEA協調放出）でアクセスが伸びる**
- 公的データなのでソース信頼性が担保しやすい
- 競合（gontarobee?）が情報過多なら、**Minimal路線で差別化する戦略**そのものは妥当

### 4.2 Built-by-AI の透明性
- 「AIで作った」と明示するのは**ネクストラボの正直さブランディング**になる
- 同時に「短期間で公開できた」事実そのものがAI実装力の証拠

### 4.3 政府データを一次ソースに使う設計
- 法的障壁ゼロ（政府標準利用規約 = CC BY 4.0互換）
- 出典明記でデータ品質を担保しつつ、運用責任を最小化

### 4.4 OGPを単一バリアントに統合した判断
- v1.0の3バリアント（neutral/caution/positive）は分岐ロジックが煩雑
- **タンクゲージ付き1枚に集約**は実装簡素化として妥当

### 4.5 4タブ構造の骨格
- カウンター / タンカー / ものさし / about の4分割は、滞在時間と SEO 流入を稼ぐ構造として悪くない

## 5. 筋が悪い点（再考すべき判断）

### 5.1 🔴 「リアルタイム」の根本矛盾
- コンセプト: **リアルタイム可視化**
- データ実体: **月次・前々月末（最大2ヶ月遅延）**
- これは**設計の根幹がデータ実態に基づいていない**ことを意味する
- 「リアルタイム」を売りにすれば景表法リスク、撤回すれば差別化点が消える
- → **コンセプトを「最新の備蓄が一目で分かる」など実態整合の表現に置き換える**べき

### 5.2 🔴 Minimalと情報追加の方向性分裂
- v1.0→v2.0で**3区分内訳とサイドメトリクスを削除**（Minimal化）
- 同時に**タンカー・ものさしの2セクションを追加**（情報追加）
- 「Minimalで静かに」と「シェアで注目を取る」が同居して破綻
- → **どちらかに振り切るか、Minimalの定義を「ノイズ削減」から「フォーカス設計」に変える**

### 5.3 🔴 取れる情報を捨てている
- 経産省データは**3区分内訳が公開されている**のに削除
- ニュース性が高いのは「内訳の動き」（IEA協調放出は国家備蓄、産油国共同備蓄の動きなど）
- **総日数だけだと「数字が動いた理由」が読めず、ユーザーがニュース時に開いても情報がない**
- → 折りたたみで内訳を表示するなど、削除でなく**情報設計の工夫で復活**させるべき

### 5.4 🔴 ドメイン理解の浅さがUXに直結
- 対象港5つに**喜入が抜けている**（中東VLCCの国内唯一の中継基地）
- 「300日」という根拠不明の基準値
- 「世界第5位」が古い（実際は第7位）
- 「314万bbl/日」が出典統一されていない
- → 一次情報の追加調査と数値の出典明記を**着手前**に行う

### 5.5 🟡 Datalastic €199のオーバーコミット
- 月額の約9割を1機能が占有
- **設計書末尾に「v2以降で導入も可」「政府統計で代替可能」と書かれているのに F-06 を「必須」**にしている矛盾
- aisstream.io（無料）で代替可能なのに高額APIを契約
- → MVPは無料代替に切り替え、Datalasticは精度差を実証してから判断

### 5.6 🟡 真の目的（リード獲得）に対するファネル設計の弱さ
- aboutタブの最後にCTAボタン1つのみ
- nextlab.jp に外部遷移するためコンバージョン計測が分断
- 「AI開発のご相談」が抽象的で、来訪者の課題と接続していない
- → CTA設計は**真の目的に直結する最重要要素**なのに薄い

### 5.7 🟡 Built-by-AIの両刃性に無策
- 「実装の99%をAIが書きました」は信頼を生む反面、**「AIだから精度が怪しい」と読まれるリスク**
- 公的データを扱うサイトでこれは致命的
- → 「実装はAI / データは経産省一次ソース」のような**信頼性の切り分け**コピーが必要

### 5.8 🟡 設計書がv1.0参照に依存
- 「アーキテクチャはv1.0参照」「実装コードはv1.0参照」で**v2.0単体では実装着手不可**
- 引き継ぎ資料としての完結性が低い
- → 差分マトリクスを別途作るか、v2.0統合版を作る

## 6. 筋を通すための再設計提案（3案）

### Plan A: リード獲得ショーケースに振り切る（推奨）

**目的を D（リード獲得）に固定**し、A/B/C は手段として最適化。

```
コンセプト: 「日本の石油備蓄、ひと目で。」
            → リアルタイム表記は使わず「最新値」で勝負

トップ:     総備蓄日数 + 3区分内訳の小さな帯（折りたたみ可）
            タンクゲージは法定義務160日 or 直近実績247日基準
            データ更新日・出典バッジを常時併記

タンカー:   aisstream.io（無料）で日本接近タンカーの隻数・推移
            喜入・水島・鹿島・苫小牧を必ず含む

ものさし:   SEO流入の入口として強化（語源説は「諸説あり」表記）

about:      「実装はAI / データは経産省一次ソース」併記
            CTAは2段構え:
              - 同一ドメイン内に簡易フォーム（メール+課題1行）
              - その後 nextlab.jp/contact へ詳細フォーム誘導
            UTM必須

月額: ~$25 (Vercel + ドメイン)
```

筋が良い理由:
- 真の目的に従属する形で全機能の優先度が決まる
- データ実態と整合（リアルタイム表記の撤廃）
- 取れる情報（3区分内訳）を捨てない
- コストが10分の1
- リード獲得計測が同一ドメインで完結

### Plan B: バイラルメディアに振り切る

**目的を C（流入・拡散）に固定**。シェアされる体験を最優先。

```
特徴:
- OGPを「インパクト型」に最適化（緊張感のあるビジュアル）
- 「いま備蓄日数 X日（前月比 ±N日）」など差分強調
- 中東情勢・IEA放出ニュースとの連動表示
- タンカー隻数・「日本まで何日」など物理スケール訴求
- ものさしを大幅拡充してSEOで原油キーワードを取りに行く
- AIS API €199 を投下する判断もありうる

リスク:
- 厳密性が下がるとブランド毀損
- バイラルは運要素が大きい
```

### Plan C: 公的ダッシュボードに振り切る

**目的を A（情報提供）に固定**。経産省データのベストUIを目指す。

```
特徴:
- 3区分内訳・基準値・推移を高密度に表示
- 出典・データ基準日を強調
- API公開（開発者向け）で他サイトに数値を提供する立場を取る
- リアルタイムは謳わず、月次更新・透明性重視
- Built-by-AI 表記は控えめ（信頼性優先）

リスク:
- リード獲得には繋がりにくい
- ネクストラボのビジネス目的との接続が弱い
```

---

## 7. 推奨: Plan A から始め、トラフィックを見て B/C を検討

**Plan A は実装コストが最小で、真の目的（リード獲得）に直結し、データ実態と整合する**。

判断基準:
- リード獲得を目的にするなら → Plan A
- バイラル・PR が目的なら → Plan B（ただし運要素大）
- 業界貢献・信頼性が目的なら → Plan C

設計書はこの選択をしないまま機能を盛っているため、**「何を作っているか分からない」感覚が生じる**。

## 8. 着手前に答えるべき3つの問い

1. **このサイトの最優先KPIは何か？**
   - 問い合わせ件数 / セッション数 / シェア数 / 滞在時間 / SEO流入 のどれを最大化するか
2. **「リアルタイム」表記は維持するか撤回するか？**
   - 維持なら景表法対応の注記文言を確定、撤回ならコンセプトコピーを書き直し
3. **AIS API €199 を初日から契約するか？**
   - 月次精度で十分なら不要。必要性の根拠（KPI上の効果見込み）を言語化

この3つに答えれば、機能優先度・コピー・コストが自動的に決まる。
