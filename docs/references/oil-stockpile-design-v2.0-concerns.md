## 0. 総括（BLUF）

v2.0は「Minimal化」と「タンカー・ものさしの追加」という方向性の異なる2つの変更を同時にかけている。前者は情報削減、後者は情報追加で、**設計上の整合がやや曖昧**である。特に以下3点は着手前に決着しておきたい。

1. **「リアルタイム」表記とデータ実体の乖離**（景表法リスク・信頼性損失）
2. **AIS APIの月€199を v2.0 MVP 時点で投下する妥当性**（コストの9割を1機能が占有）
3. **3区分内訳とサイドメトリクスを削除した結果、「総備蓄日数」単独で十分な情報価値を持つかの検証不足**

## 1. 戦略・事業判断レベルの懸念

### 1.1 🔴 Minimal化と機能追加の方向性が衝突している

v1.0では「総備蓄＋3区分内訳＋サイドメトリクス」と情報リッチ。v2.0で内訳・サイドメトリクスを削除して「総備蓄日数のみ」にした一方、タンカー・ものさしという**別タブ**で情報を増やしている。

- 結果として「カウンタータブはMinimal、他タブはリッチ」というチグハグな体験になる懸念。
- 「Minimalで事実を静かに伝える」というコンセプトと、「タンカーセクションで注目度を取る」という目的は両立しうるが、**そのストーリーが設計書内で接続されていない**。
- 推奨: トップカウンターの「Minimal」が何を犠牲にして何を得るのかを言語化（例: スクロール促進・滞在時間延長など）。サイドメトリクス削除の根拠を「ノイズだから」だけでなく**ユーザー行動仮説**で説明する。

### 1.2 🔴 3区分内訳削除のトレードオフ評価が不足

国家備蓄・民間備蓄・産油国共同備蓄は性格が異なる（放出ハードルや目的が違う）。総備蓄日数だけだと「なぜ動いたか」が読み取れず、**ニュース性のあるイベント（IEA協調放出など）でこのサイトを開いた人が欲しい情報**を提供できない。

- 競合gontarobeeに対する差別化が「Minimal」だけだと弱い。「事実を静かに伝える」だけならWikipediaで足りるという議論を内部でしておくべき。
- 推奨: 3区分内訳をトップから外すのは可だが、**「内訳を見る」のCTAでフォールド下に展開**するなど、削除でなく折りたたみが妥当ではないか再考。

### 1.3 🟡 「Built by AI 99%」訴求の両刃性

「実装の99%をAIが書きました」は透明性として好印象を与える反面、**「AIだから精度が怪しい」という不信に接続されるリスク**がある。これは備蓄数値という公的データを扱う本サイトでは特に大きい。

- 「99%」の数値根拠（コミット行数比？コードベース比？）も不明。
- 推奨: 「AIが書いた」と「データは経産省一次ソース」を併置して、**実装手法とデータ信頼性を切り分ける**コピーに調整。例: 「実装はAI、データは経済産業省統計（一次ソース）」。

### 1.4 🟡 nextlab.jp外部遷移CTAのコンバージョン設計

横幅フル黒ボタン → nextlab.jp/contact への遷移は強い導線だが:

- ドメインを跨ぐためGA等のセッション分断が発生。**コンバージョン計測の設計が未定義**。
- 注釈「※ ネクストラボの問い合わせフォームに遷移します」は離脱抑制と誠実性のバランス次第。
- 推奨: UTMパラメータ付与必須化、可能であれば**同一ドメイン内に簡易フォームを設置**してリードを直接取得（nextlab.jpへの誘導は二段構え）。

## 2. 法務・コンプライアンス懸念

### 2.1 🔴 「リアルタイム」「時分秒表示」と政府統計の月次更新ギャップ

経産省「資源・エネルギー統計」は月次更新。これを秒単位に割って表示する設計と思われるが:

- ユーザーが「秒単位で実際の備蓄量が動いている」と誤認するリスク → **景品表示法（優良誤認）の観点**で説明責任が生じる可能性。
- 「リアルタイム」という語は厳密にはストリーミング更新を指す。**実体は「推計値の連続表示」**。
- 推奨:
  - トップに「**月次統計を秒単位で按分表示しています**」を常時表示。
  - 「リアルタイム」を「ライブ推計」「推計連続表示」など事実に即した語に置換することを検討。

### 2.2 🟡 Datalastic AIS APIデータの再配信可否

AIS生データは公的情報だが、加工済みデータ（vessel_count、barrels_est等）の**公開・再配信が利用規約で許諾されているか要確認**。

- 「raw_response JSONB」をDBに保存するのは社内分析用なら可だが、**API経由で外部公開する場合は規約抵触の可能性**。
- 推奨: 契約時にBI/SaaS用途と再配信可否を明示確認し、設計書に明記。

### 2.3 🟡 出典・免責事項の表示強度

F-09に「常時表示」とあるが、フォントサイズ・配置・コントラストが不明。**「総備蓄日数」と同じスクリーン領域内で視認できる強度**が必要。

- 推奨: 数値直下またはタンクゲージ脇に小さく出典バッジ（例: 「経産省 / 2026-04時点」）を常時配置。

### 2.4 🟢 「barrel = 樽」「42ガロン」など語源説の断定

複数の説がある事項を断定すると、専門家からの指摘リスクあり。
- 推奨: 「〜という説があります」と断定回避。

## 3. データ精度・サイエンス懸念

### 3.1 🔴 タンカー積載量のDWT→バレル換算精度

「VLCC≈200万bbl、Suezmax≈100万bbl」は典型値だが、実際は:
- DWTは**重量**、バレルは**体積**。原油密度（API度）で換算係数が変わる（軽質油と重質油で±10%以上ずれる）。
- ladenでも積載率は100%でないことが多い（80〜95%）。
- 「laden判定」自体がAISのドラフト報告値からの推定で、誤判定がある。

→ **総積載量の誤差は±20%以上**になりうる。「××百万bbl」と桁を出すと過剰精度。

- 推奨: 「**およそ**」表記を必須化、または**隻数のみ**を主表示にして積載量は副次表示に格下げ。

### 3.2 🟡 「日本向けタンカー」の判定ロジック

- AISの destination フィールドは船員手入力で**未更新・誤入力が多い**ことで知られる（業界常識）。
- 5港（横浜/千葉/川崎/四日市/堺泉北）のみで日本向けをカバーするのは不足:
  - 主要原油輸入港の苫小牧・室蘭・水島・徳山下松・大分・喜入などが抜けている。
  - 特に**喜入（鹿児島）はJX ENEOSの主要受入基地**で、ここを外すと相当数の取りこぼし。
- 推奨: 主要原油受入港リストを**石油連盟・経産省資料で再洗い**してから対象港を確定。

### 3.3 🟡 「300日対比%」の基準値

- IEA基準（90日）、日本の備蓄目標（国家70日＋民間70日＋産油国共同を含めて約235日相当）など複数基準が併存。**300日の根拠が不明**。
- 「いま215日／300日」のように表示すると、**未達であるかのような誤認**を生む可能性。
- 推奨: 基準を「IEA基準（90日）」「日本の備蓄目標」「過去最大値」など複数提示するか、ゲージ最大値の根拠を脚注で明示。

### 3.4 🟡 石油のものさしの数値出典欠落

「日本の1日の消費 314万bbl/日」「世界第5位」「VLCC約340隻分」など具体数値の**出典・更新時点が未記載**。
- これらは経産省・IEA・BP統計などで毎年変動する。**静的に書くと2〜3年で陳腐化**。
- 推奨:
  - 各数値に「（出典: BP Statistical Review 2025）」のような明記。
  - 主要数値はDB化して年次更新できる構造を検討。

### 3.5 🟢 「ガソリン1Lの内訳」の経時変化

「約4割が原油、約6割が税金等」は原油相場変動で頻繁に変わる。固定文言は危険。
- 推奨: 「およそ半分が税金等」など解像度を落とすか、最新値を別途取得して動的更新。

## 4. 機能・UX懸念

### 4.1 🟡 OGPキャッシュ無効化戦略

- `/api/og?v={version}`の `v` を**月次UUID**で更新する設計だが、月内に備蓄値が大きく動いた場合（IEA協調放出等）にOGPが古いまま表示される。
- SNS側のキャッシュ（X Card Validator、Facebook Debugger）は v 変更だけでは再取得されない場合がある。
- 推奨:
  - 備蓄値の月次更新に同期して `v` を更新する仕組み（更新トリガで設定値を書き換える）。
  - イベント発生時に手動でcache_bust できるエンドポイントを準備。

### 4.2 🟡 タンクゲージの色閾値とアクセシビリティ

「緑→黄→赤」の閾値設定が未定義。さらに:
- **色のみで残量を伝えるとWCAG違反**（色覚多様性配慮不足）。
- 推奨: 色＋数値＋パターン（縞模様等）の併用。閾値はIEA90日／法定備蓄日数を境界に。

### 4.3 🟡 4タブ構成のモバイル動線

- スマホでタブが4つ並ぶとタップ領域・テキスト省略の問題。
- 「石油のものさし」のような長いラベルがどう収まるか不明。
- 推奨: タブのアイコン化検討、またはスワイプ操作のサポート。

### 4.4 🟢 シェアテンプレートのバリエーション欠如

「日本の石油備蓄、いまXXX日分。」のみだと**シェアした全員が同じ文面**でタイムラインが埋まる → 拡散効率低下。
- 推奨: 数日分のテンプレローテーション、またはユーザー入力可能フィールド。

### 4.5 🟢 「最短到着予定」のETA信頼性

AISのETAは船員手入力で精度が低い（業界では「ETAは目安」と言われる）。
- 「最短到着予定」と断定すると信頼性に傷。
- 推奨: 「予定（船舶報告ベース）」と注記。

## 5. 技術・運用懸念

### 5.1 🔴 Datalastic AIS API €199のMVP投下判断

月額コストの**約9割をこのAPIが占める**。設計書末尾に「v2以降で導入も可」「政府統計ベースの推計値で代替可能」と書かれており、**v2.0でこのAPIを必須にするか曖昧**。

- 仮に「v2.0 = リリース版」だとしても、ローンチ初期からトラフィックが少ない時期に€199は重い。
- 推奨:
  - **Phase 1: 政府統計ベース推計値**でリリース（タンカーセクションは「準備中」または推計表示）。
  - **Phase 2: トラフィック・PR効果の実績**を見てAIS API導入判断。
  - 設計書の「F-06必須」を「Phase 2必須」に変更を検討。

### 5.2 🟡 Cronジョブのエラーハンドリング

`{ "schedule": "0 21 * * *" }` のみ記載。以下が未定義:
- API失敗時のリトライ戦略（指数バックオフ？回数？）。
- 連続失敗時のアラート（Sentry連携？Slack通知？）。
- データ取得失敗時のフロント表示（「最終取得: N時間前」など）。
- snapshot_date UNIQUEなので、**再取得時のUPSERT or SKIP**ポリシーが不明。
- 推奨: vercel.jsonとは別に**運用Runbook**を用意。

### 5.3 🟡 tanker_snapshots テーブル設計

```
raw_response JSONB
```
- AISのレスポンスは1日数百〜数千レコードになる可能性。**1年で容量肥大化**。
- Supabase Freeプラン（DB 500MB）の制限に抵触するリスク。
- 推奨:
  - raw_response の保存期間ポリシー（例: 30日でサマリのみ残しBLOB削除）。
  - インデックス: `snapshot_date DESC` 必須。
  - ロールアップ用の `tanker_snapshots_monthly` 派生テーブル検討。

### 5.4 🟡 Vercel Pro $20プランの制約

- Edge Functions / Cron / Bandwidth の上限を超えるか未検証。
- OGP画像生成は @vercel/og で実行時間がかかる → Edge Function実行時間制限。
- 推奨: ローンチ前にVercel Analyticsで負荷予測、Bandwidth見積。

### 5.5 🟡 Supabase Freeプランの制限

- DB 500MB / 帯域 5GB / API呼び出しは**1秒あたりリクエスト数の制限**あり。
- バズった瞬間にレートリミット → 503エラー多発のリスク。
- 推奨: フロント側で**ISR（Incremental Static Regeneration）**を厚く設計し、Supabase直叩きを最小化。

### 5.6 🟢 監視・アラート設計の不在

- データ取得失敗・APIエラー・想定外の値（前日比±50%等）の検知方法が未定義。
- Sentry Free枠だけでは業務メトリクスは取れない。
- 推奨: 簡易ヘルスチェックエンドポイント（`/api/health`）と、Vercel Cronで日次実行する**自己診断ジョブ**。

### 5.7 🟢 為替変動

AIS API €199 は$215想定だが、為替次第で$240超えもありうる。コスト想定は**±15%バンド**で持つ。

## 6. アーキテクチャ・設計書品質の懸念

### 6.1 🟡 v1.0設計書への依存

「アーキテクチャはv1.0参照」「実装コードはv1.0参照」とあり、**v2.0設計書単体では実装着手できない**。
- 推奨: v1.0で削除された箇所（3区分内訳、サイドメトリクス）と新規追加箇所の差分マトリクスを別途整理。

### 6.2 🟡 削除機能の処理が曖昧

「3区分内訳」「サイドメトリクス」を削除→「将来検討」とあるが:
- DB側のスキーマやAPI設計まで削除するのか、UIだけ非表示か。
- 「将来検討」の判断基準（KPI？期間？）が不明。
- 推奨: 削除はUIのみ（データ層は維持）で、フィーチャーフラグで切り替え可能にすると将来戻しやすい。

### 6.3 🟢 国際化（i18n）

`Japan oil stockpile`は海外メディアにシェアされる可能性が高いトピック（IEA絡み等）。英語ページがないと拡散の上限に当たる。
- 推奨: 最低限OGPの英語版バリアントを準備。
