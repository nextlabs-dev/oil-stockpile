## 0. 結論サマリ（BLUF）

3つの重要な発見:

1. **gontarobee は「枯渇カウンター」**で、設計書v2.0 が想定していた「現在値カウンター」とは**コンセプトが根本的に違う**。「煽り型」という設計書の評価は正しい。

2. **🔴 経産省は日次速報PDFを公開していた** — 前回までの私の調査は**月次データのみと判定したが誤り**。`oil_daily.pdf` が毎日更新されており、データ時点は **2〜3日前**。これは v2.0 の「リアルタイム表示」評価を覆す重大な情報。

3. **gontarobee は完全な静的サイト + GitHub Pages 無料運用**。設計書 v2.0 の月額 $240 想定に対し、**月額 $0**で同等以上の機能を実現している。

## 3. 🔴 重大な発見: 経産省の日次速報PDFが存在する

### 私の前回までの調査の誤り
前回の feasibility 調査では:
> 「公表頻度: 月1回・毎月15日前後」
> 「データ基準日: 公表月の前々月末」
> 「リアルタイム表示は構造的に不可能」

と結論したが、**これは月次の『石油備蓄の現況』を見ていただけで、日次の速報PDFの存在を見落としていた**。

### gontarobee のソースコードから判明した事実
`reserve-chart.js` の `RESERVE_HISTORY` 配列に**38日分の日次データ**が直書きされている:

```javascript
{ published: '2026-03-17', asOf: '2026-03-14', total: 242, ... },
{ published: '2026-03-18', asOf: '2026-03-15', total: 241, ... },
{ published: '2026-03-19', asOf: '2026-03-16', total: 241, ... },
...
{ published: '2026-04-23', asOf: '2026-04-20', total: 215, ... },
```

### 日次速報PDFのURL
```
https://www.enecho.meti.go.jp/statistics/petroleum_and_lpgas/pl001/pdf-oil-res/oil_daily.pdf
```
- 同一URLが**毎日更新**されている（gontarobee README に明記）
- データ時点 = **公表日の2〜3日前**（例: 4/23公表 → 4/20時点）
- 国家/民間/産油国共同の**3区分内訳も日次で含まれる**

### 設計書v2.0への含意（評価修正）

| 項目 | 前回までの評価 | 修正後 |
|---|---|---|
| データ更新頻度 | 月次（最大2ヶ月遅延） | **日次（2-3日遅延）** |
| 「リアルタイム時分秒」表記 | 構造的に不可能・景表法リスク大 | **日次値からの按分なら相当現実的**。リスクは大幅減 |
| 3区分内訳取得 | 月次のみ | **日次で取得可能** |
| F-01 リアルタイムカウンター | 過剰精度 | **データ実態とほぼ整合する設計に変更可能** |

これは設計書v2.0 を**大幅に救済する情報**。前回までの「リアルタイム表記は撤回すべき」という指摘は**過剰な修正提案だった**ことになる。

## 4. 技術実装の詳細

### データ取得・更新
**完全に手動**:
- 毎朝、開発者が `oil_daily.pdf` を目視確認
- `script.js` の `RESERVE_DAYS`、`REFERENCE` を書き換え
- `reserve-chart.js` の `RESERVE_HISTORY` 配列に1行追加
- `index.html` のデータカード値を修正
- git commit && push → GitHub Actions が自動デプロイ

### カウントダウン計算ロジック（`script.js`）
```javascript
const dailyDraw = sc.importLoss * (1 - sc.saving);
const effectiveDays = RESERVE_DAYS / dailyDraw;
const depletionDate = new Date(REFERENCE.getTime() + effectiveDays * 86400000);

// 経過時間を引いて残量計算
const elapsed = (now - REFERENCE) / 86400000;
const consumed = elapsed * dailyDraw;
const remaining = Math.max(0, RESERVE_DAYS - consumed);
```
1秒ごとに `setInterval(updateCountdown, 1000)` で再計算。

### グラフ実装（`reserve-chart.js`）
- 720×260 SVG、純Vanilla JS で path/circle/text を生成
- マウスホバー/フォーカスで詳細表示
- 折れ線 + 点 + 数値テーブル併設
- アクセシビリティ対応（`role="button" tabindex="0"` で点をフォーカス可能）

### SEO 対応
- `<meta description>`、OGP、Twitter Card 完備
- `sitemap.xml` + `robots.txt` 設置
- guide/index.html で SEO 用読み物を別途配置
- Google Search Console 登録手順を README で整備

### プライバシー・規約
- `privacy-policy.html`（21.5KB と充実）
- 「非公式の可視化ツールです」と明記
- 「速報値は確報と差が出る場合があります」「簡易モデル」など**透明性が高い注釈**

## 6. 設計書 v2.0 への新たな示唆

### 6.1 「リアルタイム」表記の評価を緩和できる
- 経産省 `oil_daily.pdf` が**日次更新・2-3日遅延**である事実から、
- F-01 の「時分秒のリアルタイム」は、日次値の按分アニメーションとして**精度上ほぼ正当化される**
- 景表法リスクは「月次値を秒按分」と書いた前回主張より**大幅に低い**
- 注記は「最終更新からの推計値」で十分

### 6.2 3区分内訳を削除する根拠が弱まる
- 日次データに国家/民間/産油国共同の内訳が含まれる
- gontarobee はこれを4枚カードで表示
- v2.0 の「Minimal化のため削除」より、**折りたたみ表示で活かす**方が情報価値が高い

### 6.3 シナリオ機能の追加検討
- gontarobee の最大の独自価値は「もしホルムズ海峡が封鎖されたら」のシナリオ計算
- 設計書 v2.0 にはシナリオ機能がない
- ニュース性のあるイベント（中東情勢悪化など）でアクセスが伸びる時、**シナリオ機能こそが滞在時間とシェアを生む**
- 「Minimal」を維持しつつ、シナリオを副次タブとして追加する選択肢

### 6.4 AIS API €199 の必要性がさらに薄まる
- gontarobee は AIS なしで月15万PV〜（Search Console SEO 整備の充実度から推測）
- 「タンカーセクション」がなくとも備蓄カウンターとして十分機能している実例
- v2.0 がタンカー機能を「必須」と位置づける根拠がさらに弱い

### 6.5 ホスティング戦略の再考
- gontarobee は静的サイト + GitHub Pages で完結
- v2.0 が Vercel Pro + Supabase + Cron + AIS API を選んでいるのは過剰スペックの可能性
- **Phase 1 を静的サイト + GitHub Actions（月次PDF取得 → JSON生成 → push）でも実装可能**
- フロントの動的部分（カウントダウンアニメーション）はクライアント側JSで完結する

## 7. 学べる実装テクニック

### 7.1 SVGの手書きでチャートライブラリを回避
`reserve-chart.js` は **300行程度で折れ線グラフ + テーブル + ホバー詳細を実装**。Recharts や Chart.js を入れる必要なし。バンドルサイズ最小化。

### 7.2 透明性の高い注釈設計
```
※ 上の「日」は「いまから枯渇予測時刻までの残り日数」（下の時・分・秒とセット）です。
公表のデータ時点から今日までの経過も、簡易モデルで備蓄の消費に織り込んでいます。
下の「総備蓄量」は公表その時点のスナップショットのため、数字が一致しないことがあります。
```
**ユーザーが誤解しそうな箇所すべてに先回りで注釈**を置く。これが「煽り型」でありながら炎上していない理由。

### 7.3 完全静的でカウントダウンを実現する設計
- データは JS定数として埋め込み
- 計算はすべてクライアント側
- サーバAPI不要 → コスト$0、スケール無限
- v2.0 が Supabase を選んでいる理由（過去履歴の蓄積）はあるが、**月次レベルなら静的JSON配信で代替可能**

## 8. 結論

### gontarobee は明確な「煽り型カウンター」で、設計書 v2.0 のMinimal路線と競合しない
- 同じテーマ・同じデータでも**ポジションが違う**
- v2.0 の差別化戦略（Minimal）は引き続き妥当

### しかし、設計書 v2.0 の評価をいくつか修正すべき
- 🔴 **「リアルタイム表記は撤回」は過剰な修正提案**だった。日次速報PDFの存在で、時分秒表示の正当性は大幅に上がる
- 🟡 **3区分内訳の削除はやはり再考すべき**。日次で取れるなら活かす方が良い
- 🟡 **シナリオ機能の追加を検討**。ニュース性のあるイベントでバズる仕掛けとして強力
- 🟢 **ホスティングを Vercel → GitHub Pages へダウングレードする選択肢**。月額$0で同等機能が実現できる

### 真の競合は「煽り型」ではなく「コスト構造」
- gontarobee が **$0 で公開している**事実は、v2.0 の **$240/月想定** にとって脅威
- リード獲得KPIが $240/月のコストを正当化できるかが問われる
- aboutタブのCTAから **月N件以上の有効リード** を取れる前提が必要
