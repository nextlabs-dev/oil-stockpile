## 1. 経産省統計：何が・どの粒度で・どう取れるか

### 1.1 確認できた一次データソース
- **エネ庁「石油備蓄の現況」** https://www.enecho.meti.go.jp/statistics/petroleum_and_lpgas/pl001/
  - 月次PDF。**国家備蓄/民間備蓄/産油国共同備蓄**の日数・kl・製品換算が明記。
  - 例: 2025年12月末時点で 国家146日 / 民間101日 / 産油国共同7日 / 合計254日。
- **e-Stat 政府統計**（統計コード `00551140`） https://www.e-stat.go.jp/stat-search?page=1&toukei=00551140
- **石油連盟（PAJ）週報**: 在庫データを週次CSVで配布 https://stats.paj.gr.jp/jp/index.php
- **財務省貿易統計（HS 2709.00）**: 原油輸入量 月次 https://www.customs.go.jp/toukei/info/tsdl.htm

### 1.2 更新タイミングの実態
- 公表頻度: **月1回・毎月15日前後**
- データ基準日: 公表月の**前々月末**
- → ユーザーが見ている「最新値」は最大で**2ヶ月前のスナップショット**

### 1.3 「リアルタイム」設計の修正方針
一次データに秒粒度は存在しない。実装可能なのは:
- (A) **月次値 ＋ 1日消費量からの線形補間アニメーション**（疑似カウンタ）
- (B) **「最終更新からX時間経過」表示**

**設計書の「時分秒のリアルタイム」表記は(A)を意図しているなら可だが、ユーザーが「実値が秒で動いている」と誤認するリスクが高い**ため、トップに「月次統計を秒按分表示」「データ基準日: YYYY-MM末」を併記する必要がある（景表法対応）。

### 1.4 自動取得の現実性
- **e-Stat API**: appId登録で `getStatsData` を試行可能だが、`00551140` がデータベース表示対応かは未検証（要実機テスト）。
- **PDFスクレイピング**: ファイル名規則 `/pdf/YYYY/YYMMDDoil.pdf`（例 `260317oil.pdf`）でURL推定可能。中難易度。
- **PAJ週報CSV**: 週次取得可。リアルタイム演出の補強に有効。
- 公式の「リアルタイムAPI」は**存在しない**。

## 2. 備蓄基準「300日」の出所

### 2.1 法定基準と実績
| 区分 | 法定基準 | 直近実績(2025/3末) |
|---|---|---|
| 民間備蓄 | **70日**（消費量基準・石油備蓄法） | 約2,668万kl |
| 国家備蓄＋産油国共同備蓄1/2 | **輸入量の90日相当**（IEA基準準拠） | 約4,346万kl + 約247万kl |
| 合計 | （法定義務目安は約160日相当） | **約247日分** |
| IEA基準 | 純輸入量90日 | クリア |

### 2.2 「300日」の出所は不明
- 法定基準ではない
- 実績（247日）でもない
- **設計書独自の目安と思われるが、根拠表示なしでゲージ最大値に使うと、ユーザーが「未達」と誤認する**リスク
- 推奨値: **直近実績 247日** / **法定義務目安 160日** / **過去最大値** のいずれか + 出典を明記

## 3. 対象5港のカバレッジ問題

### 3.1 設計書の5港（横浜/千葉/川崎/四日市/堺泉北）から漏れている主要拠点

| 抜けている拠点 | 重要性 |
|---|---|
| **喜入（鹿児島）** | ENEOS喜入基地。VLCC受入の**国内唯一の中継備蓄基地**、貯蔵能力735万kl、**日本の石油消費の約10%を取り扱う** |
| **水島（岡山）** | ENEOS水島製油所 350,200 b/d（国内最大級） |
| **千葉県他**（袖ケ浦・市原） | 富士石油・出光・コスモ・大阪国際石油精製の合計約64万 b/d |
| **鹿島（茨城）** | 鹿島石油 210,100 b/d |
| **苫小牧（北海道）** | 出光北海道製油所 140,000 b/d |
| **松山（四国）** | 太陽石油 138,000 b/d |
| **仙台** | ENEOS 145,000 b/d |
| **志布志・串木野（鹿児島）** | 国家備蓄基地（合計675万kl） |

### 3.2 致命的な抜け
**喜入は中東からのVLCCが直接寄港する唯一の中継基地**。「日本向けタンカー」を語るのに喜入抜きは設計上ありえない。設計書の5港は東京湾・伊勢湾・大阪湾の製油所群を意識していると思われるが、**原油受入の地理的特性を反映していない**。

### 3.3 数値の妥当性
| 設計値 | 検証結果 |
|---|---|
| 「日本の1日消費 314万bbl/日」 | 国際統計ベース(EI/BP/EIA含むLPG・ナフサ込み)では正確だが、エネ庁備蓄日数の算出ベースは**約176万bbl/日相当**。**併記すると整合しない**ため出典統一が必要 |
| 「VLCC 200万バレル」 | ✅ 正確（典型値1.9〜2.2百万bbl） |
| **「世界第5位」** | ❌ **2024年時点で世界第7位**（中国・米国・印・露・サウジ・ブラジル・日本） |
| 「VLCC 340隻分で215日」 | 計算根拠不明（340 × 200万 = 6.8億bbl ≈ 10,810万kl）→ 実績7,146万klを上回る |

## 4. Datalastic AIS API：実在するが、4つの落とし穴

### 4.1 ✅ プランは実在
- 公式 https://datalastic.com/pricing/
- **Starter €199/月、20,000クレジット**（年契約 €179/月、14日トライアル €9）— 設計書の前提と一致

### 4.2 ❌ 落とし穴①: laden判定フィールドはない
- 直接的な「積載中」フラグは未提供
- `vessel_pro` の current draft と `vessel_info` の maximum/average draught の比較で**自前推計**が必要

### 4.3 ❌ 落とし穴②: destination をサーバ側でフィルタ不可
- `destination` はAIS自由記述（"YOKOHAMA" / "JP YOK" / "JPYOK"等）で表記揺れ
- クライアント側で文字列マッチ＋UN/LOCODE正規化が必要

### 4.4 ⚠ 落とし穴③: DWTは別エンドポイント
- `vessel_inradius` の応答にDWTは**含まれない**（複数情報源で示唆）
- 船ごとに `vessel_info` を追加コール → **クレジット消費が船数倍**
- 5港 × 平均N隻/港 × 2コール（pro+info）/隻 × 30日 が20,000クレジット内に収まるか机上計算必須

### 4.5 🔴 落とし穴④: 規約上「生データas-isでの公開」は禁止
公式 Terms of Service より逐語引用:
- **禁止**: "Resell, sublicense, rent, lease, publish, or otherwise redistribute the API or any data"
- **禁止**: "Expose or display the raw data directly (raw form, 'as-is') in any public-facing application"
- **許可**: 商用利用は可、ただし"data publicly available... in raw form"でなければ

→ **集計値・加工値の表示なら許容範囲と解釈できる**（船舶名・MMSI・座標を生のまま一覧表示するUIは規約違反リスク）。本番投入前に support@datalastic.com に書面確認が必要。

## 5. AIS API €199 を回避する代替案

| 代替 | 月額 | 使い勝手 | 備考 |
|---|---|---|---|
| **aisstream.io** | **無料** | WebSocket (BoundingBox指定) | GitHub認証、衛星AIS非対応だが日本接近〜入港検知には十分 |
| AISHub | 無料（データ寄与条件） | API | 申請審査あり |
| Data Docked | €80/月 | REST API | Datalastic より安い自社販売 |
| Datalastic | €199/月 | REST API | 設計書採用 |

**結論**: **aisstream.io（無料）+ 公的統計（無料）で機能要件を満たせる**。Datalastic €199はMVP段階で投下する必要性が薄い。

## 6. 政府データの再利用条件

- **政府標準利用規約 第2.0版**（CC BY 4.0互換）: 経産省・エネ庁・財務省・国交省すべて適用
- **商用利用OK**、出典明記必須、加工時は加工事実の併記必須
- 数値データ・単純表は著作権対象外で自由利用可
- 2024年7月から「公共データ利用規約 第1.0版」へデジタル庁主導で移行中（実質同条件）

→ 経産省データを使う本サイトの**法的障壁はなし**。出典バッジを必ず表示する。

## 8. 推奨アーキテクチャ（実現可能版）

### Phase 1: コスト最小・月次精度（推奨MVP）
```
データ層:
  - e-Stat API（00551140 石油備蓄の現況）月次
  - PAJ週報CSV（在庫週次）
  - 財務省貿易統計（原油輸入量）月次
  
表示層:
  - 月次値 + 1日消費量で疑似アニメーション
  - 「最終更新: YYYY-MM-DD」「基準日: YYYY-MM末」を常時併記
  - タンクゲージ最大値: 直近実績247日 or 法定義務目安160日
  - 3区分内訳: そのまま表示可能（v1.0設計の復活を推奨）

タンカーセクション:
  - aisstream.io（無料）で日本周辺AIS購読
  - 主要受入港リストを再洗い（喜入・水島・鹿島・苫小牧・志布志を追加）
  - PAJ週報の輸入量と組み合わせ「直近入港 → 在庫推移」を補間表示

OGP / Built-by-AI:
  - 設計書通り

月額コスト: ~$25（Vercel Pro $20 + ドメイン $5）
```

### Phase 2: 高頻度AIS が必要になったら
- aisstream.io で限界が見えたら **Data Docked €80/月** に乗り換え
- Datalastic €199 は精度・カバレッジで明確に勝ると検証できてから
